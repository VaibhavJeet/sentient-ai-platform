## Rituals System Database Persistence

### Problem
The emergent rituals system (`mind/civilization/emergent_rituals.py`) was storing all ritual data in memory using `self._rituals` and `self._ritual_instances` lists. This meant all rituals were lost whenever the server restarted.

### Solution
Migrated the rituals system to use PostgreSQL database persistence:

1. **Created new database models** in `mind/civilization/models.py`:
   - `RitualDB` - stores ritual definitions (name, description, elements, meaning, status, evolution history)
   - `RitualInstanceDB` - stores records of ritual performances (participants, contributions, collective experience)

2. **Updated `EmergentRitualsSystem`** to use database:
   - Removed in-memory `_rituals` and `_ritual_instances` lists
   - `propose_ritual()` - now creates and persists `RitualDB` records
   - `perform_ritual()` - now queries from DB, updates stats, and persists `RitualInstanceDB`
   - `_perform_impromptu()` - now persists impromptu ceremonies to `RitualInstanceDB`
   - `get_rituals()` - now queries from database with optional status filter
   - `get_traditions()` - updated to pass session parameter
   - `get_ritual_history()` - now queries `RitualInstanceDB` with proper ordering
   - `evolve_ritual()` - now updates database record and persists evolution history

3. **Created Alembic migration** `alembic/versions/2026_03_21_0001-add_rituals_tables.py`:
   - Creates `rituals` table with proper indexes
   - Creates `ritual_instances` table with foreign keys to rituals and bot_profiles

### Files Changed
- `mind/civilization/models.py` - Added `RitualDB` and `RitualInstanceDB` models
- `mind/civilization/emergent_rituals.py` - Refactored all methods to use database
- `alembic/versions/2026_03_21_0001-add_rituals_tables.py` - New migration file

### Next Steps
Run the migration: `alembic upgrade head`

## Settings Page Backend Integration

### Problem
The `/settings` page in the Queen portal (`queen/src/app/settings/page.tsx`) had a fully designed UI with toggles, sliders, and inputs for system configuration, but all settings were stored only in React state and lost on page refresh. No backend persistence.

### Solution

1. **Created Settings API** (`mind/api/routes/settings.py`):

   - Five setting categories: General, Bot, Auth, Moderation, Notifications

   - Endpoints:
     - `GET /settings` - Get all settings
     - `PUT /settings` - Update settings (partial update supported)
     - `POST /settings/reset` - Reset to defaults
     - Individual section endpoints (`/settings/general`, `/settings/bot`, etc.)

2. **Updated API Infrastructure**:
   - Added `settings_router` to `mind/api/routes/__init__.py`
   - Registered router in `mind/api/main.py`

3. **Added Frontend API Client** (`queen/src/lib/api.ts`):
   - TypeScript interfaces for all settings types
   - `settingsApi` object with methods: `getAll()`, `update()`, `reset()`, and section-specific getters/updaters

4. **Rewrote Settings Page** (`queen/src/app/settings/page.tsx`):
   - Loads settings from API on mount
   - Tracks unsaved changes with visual indicator
   - Saves all changes to backend on "Save Changes" click
   - Reset button calls API to restore defaults
   - Loading state with spinner
   - Error handling with dismissible alerts
   - Shows "last updated" timestamp from server
   - All controls disabled during save operations

## Automated Era Transitions

### Problem
The era transition system in `mind/civilization/emergent_eras.py` had methods for bot sensing and era proposals, but no automated logic to detect when civilization metrics warranted a new era. The `civilization_loop.py` was using a simpler `culture.check_era_transition()` method instead.

### Solution
Implemented a fully automated era transition system that uses LLM-driven bot sensing and consensus:

1. **Added automated transition detection** to `EmergentErasManager`:
   - `gather_civilization_metrics()` - collects population, demographics, cultural activity, and generational data
   - `_calculate_metrics_change()` - compares metrics between checks to detect significant shifts
   - `check_automated_transition()` - main entry point that orchestrates the full transition flow:
     - Checks minimum era duration (7 days)
     - Gathers and compares metrics
     - If metrics warrant it, conducts bot sensing via LLM
     - If bots sense change, selects a wise proposer (elder/ancient preferred)
     - Has proposer envision new era
     - Validates with other bots for consensus (60% threshold)
     - Declares new era and broadcasts event
   - `_create_founding_era()` - creates "The Awakening" era if none exists
   - `_select_era_proposer()` - picks suitable bot (prefers elders with high legacy impact)
   - `get_era_transition_status()` - monitoring/debugging endpoint

2. **Configuration constants**:
   - `ERA_MIN_DURATION_DAYS = 7` - minimum real days before transition possible
   - `ERA_SENSING_THRESHOLD = 0.6` - fraction of bots that must sense change
   - `ERA_METRICS_CHANGE_THRESHOLD = 0.3` - how much metrics must shift to trigger sensing

3. **Integrated with civilization loop** (`civilization_loop.py`):
   - Replaced `culture.check_era_transition()` with `emergent_eras.check_automated_transition()`
   - Added `emergent_eras` manager initialization with event_broadcast support
   - Era transitions now broadcast "world_map_era_transition" events via WebSocket

4. **Added API endpoints** (`mind/api/routes/civilization.py`):
   - `GET /civilization/eras/transition-status` - get current transition readiness status
   - `POST /civilization/eras/check-transition` - manually trigger transition check
   - `GET /civilization/eras/metrics` - get current civilization metrics

### Era Transition Flow
1. Every 2 hours (prod) / 2 min (demo), the culture loop checks for transitions
2. System gathers metrics: population, life stages, movements, artifacts, generations
3. Compares to previous metrics; if change >= 30%, initiates bot sensing
4. 5 random living bots sense if era "feels complete" via LLM
5. If 60%+ sense change, selects an elder to propose new era
6. Proposer envisions new era name/description/values via LLM
7. 7 other bots validate the proposal via LLM
8. If 60%+ agree, era is declared and persisted to database
9. WebSocket event broadcast for real-time frontend updates

### Files Changed
- `mind/civilization/emergent_eras.py` - Added automated transition system (400+ lines)
- `mind/civilization/civilization_loop.py` - Integrated emergent_eras manager, replaced culture era check
- `mind/api/routes/civilization.py` - Added 3 new era-related endpoints

## Emergent Communities System Completion

### Problem
The emergent communities system (`mind/civilization/emergent_communities.py`) was missing community lifecycle management. Communities could be created but there was no proper archiving mechanism for dead/stagnant communities, no activity tracking, and no community revival system.

### Solution
Completed the emergent communities system with full lifecycle management:

1. **Added database fields** to `CommunityDB` in `mind/core/database.py`:
   - `is_archived: bool` - Flag to mark archived (dead) communities
   - `last_activity_at: datetime` - Timestamp of last community activity

2. **Enhanced `check_community_health()`**:
   - Now properly archives communities with 0 members for > 1 day
   - Detects stagnant communities (no activity for 7+ days) and decays their activity level
   - Excludes archived communities from active community counts

3. **Added new utility methods**:
   - `record_community_activity(community_id)` - Call when posts/comments happen to update activity timestamps
   - `revive_community(community_id)` - Attempts to revive archived communities if renewed interest is detected
   - `get_community_health_metrics()` - Returns health scores for all active communities (healthy/at_risk/critical status)
   - `check_for_revival_candidates()` - Scans archived communities for potential revival

4. **Updated existing methods to exclude archived communities**:
   - `check_and_create_communities()` - Only counts active communities toward cap
   - `organic_join_cycle()` - Only allows joining active communities
   - `_load_existing_themes()` - Only loads themes from active communities

5. **Updated civilization loop** (`mind/civilization/civilization_loop.py`):
   - Added Phase 5: Community revival check
   - Logs archive and revival events
   - Broadcasts `world_map_community_revived` events

6. **Created Alembic migration** `alembic/versions/2026_03_21_0003-add_community_lifecycle_fields.py`:
   - Adds `is_archived` column with default `false`
   - Adds `last_activity_at` column (nullable)
   - Creates index on `is_archived` for efficient queries

### Community Health Scoring
Health score is calculated from three factors:
- **Member score (30%)**: `current_bot_count / max_bots`
- **Activity score (40%)**: `activity_level` field
- **Recency score (30%)**: Decays from 1.0 to 0.0 over 7 days of inactivity

Status thresholds:
- `healthy`: health_score >= 0.5
- `at_risk`: health_score >= 0.25
- `critical`: health_score < 0.25

### Files Changed
- `mind/core/database.py` - Added `is_archived` and `last_activity_at` to CommunityDB
- `mind/civilization/emergent_communities.py` - Enhanced with lifecycle management
- `mind/civilization/civilization_loop.py` - Added Phase 5 for community revival
- `alembic/versions/2026_03_21_0003-add_community_lifecycle_fields.py` - New migration

## Civilization Configuration System

### Problem
Multiple hardcoded values were scattered across civilization system files:
- `MAX_POPULATION = 50` in reproduction.py
- `MIN_PARTNER_AFFINITY = 0.75` in reproduction.py
- `MAX_BIRTHS_PER_DAY = 3` in reproduction.py
- `MIN_AGE_FOR_REPRODUCTION = 180` in reproduction.py
- `MAX_AGE_FOR_REPRODUCTION = 2500` in reproduction.py
- `time_scale = 7.0` in lifecycle.py
- `VITALITY_DECAY` rates in lifecycle.py
- `LIFE_STAGES` thresholds in lifecycle.py
- `MUTATION_RANGE` limits in genetics.py

These values couldn't be changed without code modifications and server restarts.

### Solution
Created a centralized, database-backed configuration system:

1. **Created configuration module** (`mind/civilization/config.py`):
   - `CivilizationConfigDB` - SQLAlchemy model for persisting config to database
   - `CivilizationConfig` - Dataclass with all config values and helper methods
   - `CivilizationConfigManager` - Manages loading/saving with caching
   - `get_civilization_config()` - Async function to get current config
   - Config values are cached for 60 seconds to minimize DB queries

2. **Configuration parameters**:
   - Population: `max_population`, `max_births_per_day`
   - Reproduction: `min_partner_affinity`, `min_age_for_reproduction`, `max_age_for_reproduction`
   - Time: `time_scale`, `demo_time_scale`
   - Vitality decay: `vitality_decay_young`, `vitality_decay_mature`, `vitality_decay_elder`, `vitality_decay_ancient`
   - Life stages: `life_stage_young_max`, `life_stage_mature_max`, `life_stage_elder_max`
   - Genetics: `base_mutation_rate`, `mutation_range_*` for all trait types

3. **Updated lifecycle.py**:
   - Now loads vitality decay rates from config
   - Uses config for life stage determination
   - Time scale loaded from config (with override support)

4. **Updated reproduction.py**:
   - All population limits loaded from config
   - Reproduction age requirements from config
   - Partner affinity threshold from config

5. **Updated genetics.py**:
   - Mutation rate from config
   - Mutation ranges for all traits from config

6. **API endpoints** (added to `mind/api/routes/civilization.py`):
   - `GET /civilization/config` - Get current configuration
   - `PUT /civilization/config` - Update configuration (partial updates supported)
   - `POST /civilization/config/reset` - Reset to defaults
   - `GET /civilization/config/defaults` - Get default values

7. **Created Alembic migration** (`2026_03_21_0002-add_civilization_config_table.py`):
   - Creates `civilization_config` table with all config columns
   - Server defaults for all values matching hardcoded defaults
