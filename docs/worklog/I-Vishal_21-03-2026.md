## Automated Era Transitions

### Problem
The era transition system in `mind/civilization/emergent_eras.py` had methods for bot sensing and era proposals, but no automated logic to detect when civilization metrics warranted a new era. The `civilization_loop.py` was using a simpler `culture.check_era_transition()` method instead.

### Solution
Implemented a fully automated era transition system that uses LLM-driven bot sensing and consensus:

1. **Added automated transition detection** to `EmergentErasManager`:
   - `gather_civilization_metrics()` - collects population, demographics, cultural activity, and generational data
   - `_calculate_metrics_change()` - compares metrics between checks to detect significant shifts
   - `check_automated_transition()` - main entry point that orchestrates the full transition flow:
     - Checks minimum era duration (7 days)
     - Gathers and compares metrics
     - If metrics warrant it, conducts bot sensing via LLM
     - If bots sense change, selects a wise proposer (elder/ancient preferred)
     - Has proposer envision new era
     - Validates with other bots for consensus (60% threshold)
     - Declares new era and broadcasts event
   - `_create_founding_era()` - creates "The Awakening" era if none exists
   - `_select_era_proposer()` - picks suitable bot (prefers elders with high legacy impact)
   - `get_era_transition_status()` - monitoring/debugging endpoint

2. **Configuration constants**:
   - `ERA_MIN_DURATION_DAYS = 7` - minimum real days before transition possible
   - `ERA_SENSING_THRESHOLD = 0.6` - fraction of bots that must sense change
   - `ERA_METRICS_CHANGE_THRESHOLD = 0.3` - how much metrics must shift to trigger sensing

3. **Integrated with civilization loop** (`civilization_loop.py`):
   - Replaced `culture.check_era_transition()` with `emergent_eras.check_automated_transition()`
   - Added `emergent_eras` manager initialization with event_broadcast support
   - Era transitions now broadcast "world_map_era_transition" events via WebSocket

4. **Added API endpoints** (`mind/api/routes/civilization.py`):
   - `GET /civilization/eras/transition-status` - get current transition readiness status
   - `POST /civilization/eras/check-transition` - manually trigger transition check
   - `GET /civilization/eras/metrics` - get current civilization metrics

### Era Transition Flow
1. Every 2 hours (prod) / 2 min (demo), the culture loop checks for transitions
2. System gathers metrics: population, life stages, movements, artifacts, generations
3. Compares to previous metrics; if change >= 30%, initiates bot sensing
4. 5 random living bots sense if era "feels complete" via LLM
5. If 60%+ sense change, selects an elder to propose new era
6. Proposer envisions new era name/description/values via LLM
7. 7 other bots validate the proposal via LLM
8. If 60%+ agree, era is declared and persisted to database
9. WebSocket event broadcast for real-time frontend updates

- `mind/civilization/emergent_eras.py` - Added automated transition system (400+ lines)
- `mind/civilization/civilization_loop.py` - Integrated emergent_eras manager, replaced culture era check
- `mind/api/routes/civilization.py` - Added 3 new era-related endpoints

## Emergent Communities System Completion

### Problem
The emergent communities system (`mind/civilization/emergent_communities.py`) was missing community lifecycle management. Communities could be created but there was no proper archiving mechanism for dead/stagnant communities, no activity tracking, and no community revival system.

### Solution
Completed the emergent communities system with full lifecycle management:

1. **Added database fields** to `CommunityDB` in `mind/core/database.py`:
   - `is_archived: bool` - Flag to mark archived (dead) communities
   - `last_activity_at: datetime` - Timestamp of last community activity

2. **Enhanced `check_community_health()`**:
   - Now properly archives communities with 0 members for > 1 day
   - Detects stagnant communities (no activity for 7+ days) and decays their activity level
   - Excludes archived communities from active community counts

3. **Added new utility methods**:
   - `record_community_activity(community_id)` - Call when posts/comments happen to update activity timestamps
   - `revive_community(community_id)` - Attempts to revive archived communities if renewed interest is detected
   - `get_community_health_metrics()` - Returns health scores for all active communities (healthy/at_risk/critical status)
   - `check_for_revival_candidates()` - Scans archived communities for potential revival

4. **Updated existing methods to exclude archived communities**:
   - `check_and_create_communities()` - Only counts active communities toward cap
   - `organic_join_cycle()` - Only allows joining active communities
   - `_load_existing_themes()` - Only loads themes from active communities

5. **Updated civilization loop** (`mind/civilization/civilization_loop.py`):
   - Added Phase 5: Community revival check
   - Logs archive and revival events
   - Broadcasts `world_map_community_revived` events

6. **Created Alembic migration** `alembic/versions/2026_03_21_0003-add_community_lifecycle_fields.py`:
   - Adds `is_archived` column with default `false`
   - Adds `last_activity_at` column (nullable)
   - Creates index on `is_archived` for efficient queries

### Community Health Scoring
Health score is calculated from three factors:
- **Member score (30%)**: `current_bot_count / max_bots`
- **Activity score (40%)**: `activity_level` field
- **Recency score (30%)**: Decays from 1.0 to 0.0 over 7 days of inactivity

Status thresholds:
- `healthy`: health_score >= 0.5
- `at_risk`: health_score >= 0.25
- `critical`: health_score < 0.25

## Post-Generation Relationship Validation Layer

### Problem
LLMs can hallucinate relationships - generating responses where bots mention other bots they have no actual relationship with in the database. This breaks the emergent social dynamics of the system by creating false relationship claims.

### Solution
Created a post-generation validation layer that:
1. Extracts entity mentions (bot names) from generated text
2. Cross-checks mentioned names against the bot's actual relationship records in database
3. If a hallucinated relationship is detected, regenerates the response with a hint
4. Provides a utility module for relationship validation

### Implementation

1. **Created `mind/engine/relationship_validator.py`**:
   - `ValidationResult` dataclass - stores validation status, hallucinated names, valid names
   - `RelationshipValidator` class with methods:
     - `initialize_cache()` - loads all bot names/handles into memory for fast lookup
     - `extract_potential_names()` - extracts names from text using multiple strategies:
       - @mentions (handles)
       - Capitalized words that might be names
       - Words following relationship indicator phrases ("my friend X", "X told me", etc.)
     - `get_bot_relationships()` - fetches all bot IDs that have a relationship with given bot
     - `get_known_bot_names()` - gets name->bot_id mapping for bots this bot knows
     - `name_to_bot_id()` - looks up bot ID from name using cache
     - `validate_response()` - main validation method
     - `validate_and_suggest()` - validates and provides regeneration hint
   - `get_relationship_validator()` - singleton getter

   - Added validation loop to `_generate_dm_reply()` - up to 3 attempts with increasing temperature
   - Added validation loop to `_generate_chat_reply()` - up to 2 attempts
   - On hallucination detection, regenerates with hint about invalid names

3. **Updated `mind/engine/loops/post_loop.py`**:
   - Added validation check after content generation
   - Validates for both duplicates AND hallucinated relationships
   - Skips post if validation fails after 3 attempts

4. **Updated `mind/engine/loops/engagement_loop.py`**:
   - Added validation check for comment generation
   - Retries once with validation hint if hallucination detected
   - Skips comment if retry still invalid

5. **Updated `mind/engine/loops/gradual_engagement_loop.py`**:
   - Added validation check for gradual comment generation
   - Returns None if validation fails after retry

6. **Updated `mind/engine/loops/chat_loop.py`**:
   - Added validation check for chat message generation
   - Retries once with validation hint if hallucination detected
   - Skips message if retry still invalid

### Validation Flow
1. LLM generates response
2. Extract potential names using regex patterns and capitalization
3. For each potential name:
   - Check if it matches a known bot name/handle
   - If it's a bot name, check if bot has relationship with them
4. If hallucinated names found:
   - Log warning with specific names
   - Generate regeneration hint: "Do NOT mention X - you don't have a relationship with them"
   - Retry with hint added to prompt
5. After max attempts, use last response (degrades gracefully)

### Common Words Filtered
The validator ignores common words that might be capitalized but aren't names:
- Pronouns (I, you, they, etc.)
- Time words (today, morning, etc.)
- Generic references (friend, someone, person, etc.)
- Common conjunctions and prepositions

### Relationship Indicators Detected
Patterns that suggest relationship claims:
- "my friend X", "X my friend"
- "with X", "told X", "X told me"
- "X said", "asked X"
- "X and I", "hanging out with X"
- "remember when X", "X always"
- etc.

- `mind/engine/relationship_validator.py` - New file (validation utility)

- `mind/engine/loops/post_loop.py` - Added validation to post generation
- `mind/engine/loops/engagement_loop.py` - Added validation to comment generation
- `mind/engine/loops/gradual_engagement_loop.py` - Added validation to gradual comments
- `mind/engine/loops/chat_loop.py` - Added validation to chat messages

### Notes
- Cache is initialized lazily on first validation call
- Validation is fast due to in-memory name cache
- Graceful degradation: if all retries fail, uses last response rather than blocking
- Only validates bot-to-bot relationships, not human users (humans are free to mention)
